# Management Talk

Same audience and translation rules as a written status report, but **shaped for the channel** — JIRA comment, Slack post, async standup, email, or meeting talking-points. The audience reads code names but not code.

## When to invoke

- "write something for management / exec / VP / director / PM / release manager"
- "rewrite this for [non-eng audience]"
- "make this non-technical" / "less techy" / "less jargony"
- "send a slack update / standup note / email" *about a piece of engineering work*
- "executive summary" / "exec summary" / "leadership update" / "status update"
- "talking points for [meeting]" *based on an engineering update*

## Channel shapes

### JIRA comment / written status report
Full structured block. Bolded section labels. Easy to scan from the ticket page.

Building blocks: **Status / TL;DR**, **Impact**, **What broke**, **Why now**, **Owner**, **Next steps**, **Workaround**, **Risk**.

### Slack — channel post or DM
Single message, no walls of text. One **bolded TL;DR** as the first line. 2–4 short bullets. One link. ~80 words max.

### Async standup note
1–3 lines, max. Pattern: *"\<state\> \<thing\>. \<owner if not me\>. \<next\>."*

### Email — internal exec / cross-team
Subject line is half the value. Body: flowing paragraphs, 2–3 paragraphs.

### Meeting talking-points
Bullet list, max one short clause per bullet. Order is the order you'll speak in.

## Rules

- **Never invent facts** to make the rewrite cleaner.
- **Never strip a JIRA key, PR number, or customer/workload name.**
- **Never invent owners.**
- **Get sign-off before posting to JIRA.**
- **Never post to Slack, email, or any non-JIRA channel from this skill.**
- **Stay out of advocacy.**
