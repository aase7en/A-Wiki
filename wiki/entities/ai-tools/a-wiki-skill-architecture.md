# A-Wiki Universal Skill Architecture

[verified 2026-07-01]

> **Tagline:** One registry, every agent. Edit the registry, regen, commit — every agent surface updates automatically.

## Problem solved

A-Wiki grew skills across 6+ locations (skills/, agent-skills/, ~/.claude/, ~/.kilo/, ~/.codex/) with two incompatible discovery paradigms (explicit manifests vs recursive auto-discover) and no enforcement against drift. Result: agents saw different skills, adding an agent required manual multi-file edits, and duplicates accumulated.

## Solution: 5-layer contract architecture

```
LAYER 0: SKILL.md files (content) — skills/ · agent-skills/ · ~/.claude/ · ~/.kilo/ · ~/.codex/
LAYER 1: skills-registry.json  ← SINGLE SOURCE OF TRUTH (name, aliases, domain, lifecycle, status)
LAYER 2: Generators → per-agent surfaces (.kilo kilo.jsonc, .gemini settings.json, AGENTS.md tables)
LAYER 3: Agents load surfaces natively
LAYER 4: Enforcement (PreToolUse hook + pre-commit + CI)
```

**Key principle:** The registry is the only file agents edit for skill management. Generated surfaces are NEVER hand-edited — they're regenerated by `scripts/regen-skill-surfaces.py`.

## How to add a new agent (future-proofing)

Adding a new agent requires exactly 2 changes:
1. Add a generator module in `scripts/skills_registry/generators/gen_<agent>.py` exposing `render(registry)` + `filename`.
2. Add one line to the `GENERATORS` dict in `scripts/regen-skill-surfaces.py`.

Nothing else changes. The registry, drift detection, hook, CI, and all other agents' surfaces continue working.

## How to add a new skill

**Order matters (CLICK-PATH-001):** the hook blocks a Write to `SKILL.md` whose name is not yet in the registry. Register FIRST, then author.

1. **Register** — add an entry to `skills-registry.json` (`name`, `domain`, `lifecycle_phase`, `category`, `source: repo`, `path`, `agents: ["all"]`, `status: canonical`). To scaffold a draft entry: `python scripts/regen-skill-surfaces.py --bootstrap --out skills-registry.draft.json` then copy the relevant entry into the real registry.
2. **Regenerate surfaces** — `python scripts/regen-skill-surfaces.py` (so the new skill appears in generated tables/paths).
3. **Author the SKILL.md** at the declared `path` — the hook now passes because the name is registered.
4. **Commit** — the hook warns if frontmatter is incomplete; CI verifies no drift.

## On a fresh clone

A fresh `git clone` already contains `skills-registry.json` and the committed `generated/` surfaces. To verify they're in sync (they should be, byte-identical, since generators are deterministic):

```bash
python scripts/regen-skill-surfaces.py --check    # expect: no drift
```

If `--check` reports drift (e.g. a previous commit bypassed regen), run `python scripts/regen-skill-surfaces.py` (no flags) to regenerate, then commit. **Do NOT use `--bootstrap` on a clone that already has a registry** — `--bootstrap` writes a separate `skills-registry.draft.json` and never touches the real registry or surfaces (it's only for first-time registry creation on a repo with no registry).

## How to deduplicate skills

See `docs/protocols/skill-consolidation.md`. Summary:
- **True duplicate** (same content, different paths): keep canonical, record mirrors in `mirror_paths[]`.
- **Thin stub** (re-route entrypoint): mark `status: alias` + `canonical: <name>`.
- **Deprecated**: mark `status: deprecated` + `migrated_to: <name>`.
- **Near-duplicate** (same purpose, different framework): keep content, alias the surface.

The consolidation tool (`scripts/skills_registry/consolidate.py`) is idempotent — safe to run repeatedly.
